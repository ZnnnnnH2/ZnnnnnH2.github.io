ctfHub-RCE-命令注入

无过滤

使用&&连接命令，通过查看源代码找到 flag

  • ; 不管成功失败都继续下一条
  • && 前一条成功才执行下一条
  • || 前一条失败才执行下一条
  • | 把左边输出当右边输入
  • (…) 在子 shell 中执行一组命令
  • %0a 或 %0d

过滤 cat

使用 tac 代替 cat

当cat被过滤后,可以使用一下命令进行读取文件的内容

  • more:一页一页的显示的显示档案内容
  • less:与more类似,但是比more更好的是,他可以[pg dn][pg up]翻页
  • head:查看头几行
  • tac:从最后一行开始显示,可以看出tac是cat的反向显示
  • tail:查看尾几行
  • nl:显示的时候,顺便输出行号
  • od:以二进制的方式读取档案内容
  • vi:一种编辑器，这个也可以查看
  • vim:一种编辑器,这个也可以查看
  • sort:可以查看
  • uniq:可以查看
  • file -f:报错出具体的内容

过滤

  • 使用 < 重定向符代替空格，原理是将 <* 重定向到命令的标准输入
  • 使用 ${IFS} 代替空格，IFS 是环境变量，表示内部字段分隔符，默认是空格、制表符和换行符

过滤 /\

通过 ls 得知 flag 在 ./flag_is_here/flag_22183547332303.php 中

  • 使用命令127.0.0.1&&cd flag_is_here&&cat flag_22183547332303.php
  • 使用命令127.0.0.1&&cd flag_is_here&&cat f*
  • 使用命令cat flag_is_here${PATH:0:1}flag_252412887014927.php 在PATH 环境变量中取第一个字符 /

过滤关键字

  • 使用 \ 转义被过滤的字符
  • 直接使用通配符，不使用完整字符串

ctfHub-RCE-文件包含

  1. 文件包含漏洞（File Inclusion）：程序把用户提供的路径或 URL 直接传给语言的包含函数（例如 PHP 的 include/require），没有做足够的校验。
  2. 变成 RCE：攻击者把“恶意代码”放到服务器上（或让包含函数访问到带代码的资源），包含后语言解释器会执行这些代码 → 就变成了远程代码执行。

LFI

LFI（Local File Inclusion，本地文件包含）：只允许包含服务器本地的文件（比如 include($_GET[‘f’]);），如果能读到 /etc/passwd、应用日志等，可能被进一步利用。

RFI

RFI（Remote File Inclusion，远程文件包含）：如果应用允许包含远程 URL（如 http://…），攻击者能让应用直接从远端拉取并执行恶意脚本（更直接的RCE）。

  1. 确定是否允许远程包含：尝试传入 URL 或确认phpinfo 中的 allow_url_include 设置为 On。
  2. 上传恶意脚本：如果应用允许文件上传，可以上传一个包含恶意代码的脚本，使用php://input、data://等包装器。
